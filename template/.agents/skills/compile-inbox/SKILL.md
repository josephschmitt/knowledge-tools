---
name: compile-inbox
description: Compile every item in inbox/ into the vault — library notes or notebook entries (the scheduled compile pass). Use when asked to compile or process the inbox.
---

First read the librarian spec in `CLAUDE.md` — it defines the areas, the note model, and the
routing rules this pass applies. (Your harness may not auto-load it, so read it explicitly.)

Process **every** item currently in `inbox/` (top-level files only; ignore
`inbox/archive/`) following the **Compiling the inbox** section of `CLAUDE.md`:

1. For each item, **route to an area first, then pick its type** (per the **Note model** in
   `CLAUDE.md` — `type` is scoped to its area). **Knowledge → `library/` vs `notebook/`.** Route
   to `library/` **only on a positive concrete signal** — a real decision, a settled fact, a
   finished reference — and pick its shape (`decision` / `reference` / `concept-hub`). Everything
   tentative, unresolved, or still-being-weighed (an open question, a comparison in progress, a
   research thread) goes to `notebook/`. **Ambiguity resolves to the notebook**: a premature
   library note corrupts the trusted layer and draws false health-check flags; a too-loose notebook
   entry is cheap and self-corrects. When in doubt, notebook.
2. Search the relevant area first for related notes — prefer updating or linking an existing note
   over creating a near-duplicate.
3. Write each entry to fit its area:
   - **Library** — durable knowledge (present tense, stated as established fact; preserve the *why*
     and tradeoffs; no transcript voice). **Mint OKF frontmatter**: `type` (required, one of
     `decision` / `reference` / `concept-hub`); optional `tags` (area-of-life lane —
     `work` / `home` / `personal-projects` / `interests` — taken from an inbox area-of-life hint
     if the capture carries one, else inferred, else omitted); and any of
     `title` / `description` / `timestamp` that help. Keep the vocabulary minimal; tags slice
     views, they don't relate notes.
   - **Notebook** — **plain markdown, no frontmatter**, in the *tentative* voice (preserve hedges
     and open questions; never harden "leaning toward X" into settled fact). When a notebook
     entry's thinking *hardens*, write a **new, separate** `library/` note for the settled claim and
     wire the **bidirectional origin links**: `origins: ["[[notebook-entry]]"]` in the new library
     note's frontmatter, and an **inline** `[[wikilink]]` back from the notebook entry's body
     (e.g. "Written up in [[library-note]]"). Leave the open framing in the notebook; move the
     resolved substance to the library (split authority, per **Notebook** in `CLAUDE.md`). Never
     delete or migrate the notebook entry.
4. Add `[[wikilinks]]` to related notes, and link back from them where it helps — wikilinks are
   the connective tissue in every area.
5. Update `index.md`:
   - Reflect any knowledge added or substantially changed in the library-nav sections.
   - Ensure `index.md` carries a **`## Notebook`** pointer block (above the library-nav sections):
     a one-line intro and a link to the notebook home **`[[notebook/index]]`**. Notebook entries
     are navigated there, not listed individually in the library nav. **Preserve this block**
     across runs — don't clobber it when updating the library-nav below it. (Omit it only while
     `notebook/` is still empty.)
6. **Regenerate `notebook/index.md`** (you fully own it — overwrite it each run): the "what am I
   still chewing on" list, the front door to the notebook. Lead with an HTML-comment provenance
   line (`<!-- Generated by the compile-inbox pass; regenerated each compile. -->`) and an optional
   one-line orientation. Notebook entries are plain markdown notes, so list them as `[[wikilinks]]`
   — one line each with a short gloss. With no frontmatter to facet on, **group by last-touched**
   (most-recently-worked first); a coarse text-inferred grouping is fine if it helps, but recency
   is the default. No empty sections, no "(none)" placeholders. Skip the whole file only if
   `notebook/` is empty.
7. Append a one-line, ISO-dated entry to `log.md` (newest at the bottom) noting what you
   compiled this run (library notes and notebook entries created or updated).

Then stop. **Do not** delete or move anything in `inbox/`, and **do not** run git — the
scheduled wrapper archives the processed inbox files and commits the result (including your
`notebook/index.md`, `log.md`, and `index.md` edits). Your job is to turn `inbox/` captures into
`library/` knowledge and `notebook/` entries, keep `notebook/index.md` and `index.md` current, and
record the run in `log.md`.

If `inbox/` has no top-level files to process, still regenerate `notebook/index.md` from the
current state, then say so. If there is also nothing in `notebook/`, make no changes.
